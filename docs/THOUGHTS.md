# 프로젝트 구현 과정에서 정리한 생각 

## 1. 기본 원칙: 코드보다 명세를 먼저 맞춘다
- 과제 설명 문서와 API 상세 명세를 함께 보고, 충돌이 있을 때는 실제 채점 기준에 가까운 상세 API 문서를 우선으로 해석했다.
- 테스트도 구현을 설명하는 문서라고 보고, "명세 문장 -> 테스트 케이스"로 매핑하는 방식으로 작성했다.

## 2. 테스트 전략: 서비스 레이어를 비즈니스 규칙의 중심으로 본다
- 필수/가산점 API를 전부 서비스 메서드 기준으로 나눠 단위 테스트를 작성했다.
- 에러 코드는 단순 예외 발생이 아니라 정확한 `ErrorCode`를 검증했다.
- 패널티, 보증금, 공결 횟수처럼 상태 변화가 있는 로직은 전/후 상태를 같이 검증했다.

## 3. 시드 데이터 의존성에 대한 고민
- 로그인 성공 테스트를 `admin` 시드 계정이 아니라 일반 회원 계정 기준으로 변경했다.
- 이유는 단위 테스트가 시드 데이터 존재 여부에 의존하면 테스트 의미가 약해지기 때문이다.
- 대신 시드 데이터 자체는 통합 테스트로 별도 검증했다.

## 4. 테스트 코드 기준
- 테스트 코드는 모두 ai로 작성하였다.
- 함수명은 모두 한글로 바꿔 의도를 바로 읽히게 했다.
- 모든 테스트에 `given / when / then` 주석을 추가해 구조를 통일했다.
- 테스트 이름과 실제 검증 내용이 어긋나면 이름을 바꾸거나 케이스를 보강해야 한다는 기준을 세웠다.

## 6. 서버 재시작(초기화) 관점 검증
- `SeedDataInitializer` 통합 테스트를 추가해 서버 기동 시 데이터 생성을 확인했다.
- 같은 초기화 로직을 다시 실행해도 중복이 생기지 않는지(idempotent)도 검증했다.

## 7. 내가 이 프로젝트에서 유지하려는 구현 기준
- 테스트는 단순 통과가 아니라 설계 의도를 드러내야 한다.
- 명세 충돌은 무시하지 않고 문서-코드-테스트를 한 번에 정렬한다.
- 도메인 모델(Member vs CohortMember) 경계를 명확히 잡아야 보증금/출결 로직이 안정적이다.
